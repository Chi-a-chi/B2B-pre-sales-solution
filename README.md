# Industry Penetration Skill

在大厂干了几年toB解决方案,最近整理了一下过往做行业方案时的一套 B2B 行业分析方法，区别于市场上常见的分析报告，我们为客户做方案时更关注企业在行业竞争/产业变革/产业链生态/企业自身经营等方面的痛点和机会。

## 它做了什么

**先不预设你是谁**。产出的机会图景是客观的,每个摩擦节点都标清楚谁在承担、承担方强势还是弱势、可能有哪些介入者类型。你是SaaS?是咨询?是产业资本?那是下一步自己二次筛选的事,跟这一步分析无关。

**真伪痛点明着分开**。方法里内置了几条判据:多人从不同角度提到同一件事 = 结构性问题;单人反复强烈吐槽 = 可能只是个案;强势方抱怨和弱势方抱怨权重不一样;吐槽的事是会自己消失的过渡性问题还是长期存在的结构性问题。跑完给你两份清单,真痛点和伪痛点分开放。

**框架藏在问题里**。五力、PEST 这些框架都不单独列章节,而是化整为零,变成七层检查清单里的一个个具体问题。你看到的是"资金到账和票据匹配之间有没有时间差?这个时间差里的钱归谁?"——不是"请用五力分析这个行业"。框架该干的活是让你问对问题,不是让你凑报告章节。

## 两种用法

**第一种,给个行业名**。比如"分析一下预制菜 B 端"。走三层结构:先猜你真正想知道什么(动机),然后说我打算用什么角度切(框架),再展开(具体内容)。展开完强制过七层检查,漏了的补回来。最后产出一份深度分析档,归档。

**第二种,贴一堆素材**。比如你从脉脉、小红书、知乎、行业群扒了一堆用户吐槽,丢进来。它不会按素材自己的话题分类,而是把每条吐槽映射到九流 + 合规五类的坐标里,然后判断真伪,输出两份清单。

这两种模式可以互喂。素材跑出来发现某个行业有料,转头让它走第一种模式做深度分析。

## 什么时候别用

- **消费端的事别找它**。这套方法是为 B 端设计的,九流、合规、议价结构这些维度在 C 端没那么重要
- **已经锁定视角的分析别找它**。比如"银行怎么切 XX 行业"——你已经带着能力方视角了,这套方法的核心立场就被破坏了。那种需求应该是另一个 skill
- **简单事实查询别找它**。问"餐饮行业多大规模"这种,直接搜就行,不需要上方法论

## 目录结构

```
industry-penetration/
├── README.md
├── SKILL.md                 # skill 入口
├── rules/
│   ├── analysis-method.md   # 三层回答、九流、演绎法、合规必选项
│   ├── checklist.md         # 七层检查清单
│   └── handoff.md           # 对话收尾机制
└── archive/
    ├── industry-map.md      # 行业全景 + 研究价值排序
    └── deep-dives/
        ├── _template.md     # 深度分析模板
        └── _HOWTO.md        # 填写指引
```

## 一点设计上的洁癖

**方法和实例是分开的**。`rules/` 里放的是纯方法,换个项目、换个行业都能用;`archive/` 里才是具体产出。我见过太多"方法论"写着写着就跟某个具体案例粘在一起,下次换个行业又得重写。这次想从一开始就切干净。

**对话有收尾机制**。长项目跑到后面,上下文会散,关键判断会丢。`handoff.md` 定义了一个四段式的转接格式:核心结论、悬置问题、上下文关键词、规则变更清单。你说一句"转接新对话"就自动生成,下次开个新窗口从这里续上就行。
